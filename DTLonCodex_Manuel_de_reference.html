OpenAI Codex — Manuel de référence

Version de travail française, d'après la documentation officielle OpenAI Codex et la présentation documentaire DEC/VSI.

Préface

Ce manuel de référence décrit les fonctions d'OpenAI Codex. Il ne constitue pas un guide d'utilisation pas à pas. Les commandes, menus et fichiers sont mentionnés uniquement pour identifier les fonctions qu'ils exposent.

Codex est l'agent de développement logiciel d'OpenAI. Il assiste les activités de programmation, d'analyse de code, de revue, de correction d'anomaliés, de migration, de documentation, de tests et d'automatisation de travaux de développement.

Public visé

Ce manuel s'adresse aux développeurs, mainteneurs, responsables techniques, équipes de revue et administrateurs qui doivent connaître les fonctions disponibles dans Codex et les limites de chaque surface d'exécution.

Conventions

Les noms de fichiers, options, commandes, variables et identifiants sont indiqués en chasse fixe. Les termes App, CLI, IDE Extension, Web, Local, Worktree et Cloud designent les surfaces et modes Codex officiels.

1. Vue d'ensemble fonctionnelle

Codex fournit un agent interactif capable de lire un espace de travail, raisonner sur son contenu, proposer un plan, modifier des fichiers, exécuter des commandes, vérifier des résultats et rendre compte de son travail.

Les fonctions principales comprennent :

  • génération et modification de code ;
  • compréhension et explication de bases de code ;
  • revue de changements ;
  • diagnostic et correction d'erreurs ;
  • exécution de commandes locales ou distantes selon la surface ;
  • production et vérification d'artefacts ;
  • gestion de tâches parallèles ;
  • interaction avec des outils externes ;
  • automatisation de tâches récurrentes.

Codex applique les instructions du fil de discussion, les consignes de projet, les règles de configuration et les contraintes de sécurité disponibles dans l'environnement actif.

2. Surfaces d'exécution

2.1 Codex App

La Codex App est une application de bureau pour macOS et Windows. Elle fournit une expérience centrée sur les fils Codex, les projets, les worktrees, les automatisations et les fonctions Git intégrées.

Elle comprend notamment une liste de projets, des fils de discussion par projet, des modes Local, Worktree et Cloud, un terminal intégré par fil, un panneau de différences Git, des fonctions Git courantes, un navigateur intégré, la prise en charge des artefacts et la prise en charge des skills, MCP, notifications, mémoires et automatisations.

2.2 Codex CLI

Le cliént CLI fournit une interface terminale plein écran et des sous-commandes non interactives. Il permet de travailler depuis un répertoire local, de reprendre des sessions et d'exécuter Codex dans des workflows automatisés.

Les fonctions CLI comprennent le mode interactif conversationnel, le lancement avec prompt initial, la reprise de sessions précédentes, la sélection de modèle, l'affichage et la copie de sorties, l'historique de prompts, l'exécution non interactive, le serveur d'application et le mode distant.

2.3 Extension IDE

L'extension IDE place Codex dans l'éditeur. Elle fournit un contexte issu des fichiers ouverts et permet d'exécuter des tâches directement depuis le projet de l'éditeur.

Elle prend en charge le mode agent par défaut, la lecture et la modification dans le répertoire du projet, la synchronisation avec la Codex App lorsque les deux surfaces sont ouvertes sur le même projet, les commandes IDE, les commandes slash et le contexte automatique des fichiers consultes.

2.4 Codex Web et Cloud

La surface Web permet d'exécuter des tâches Codex dans des environnements cloud configures. Les environnements cloud découplent l'exécution de la machine locale et permettent des tâches parallèles ou déléguées.

Les fonctions cloud dépendent de la configuration d'environnement, de l'accès au dépôt, des politiques réseau et des intégrations actives.

2.5 Modes Local, Worktree et Cloud

Le mode Local travaille dans le répertoire courant du projet. Le mode Worktree isole les modifications dans un worktree Git séparé. Le mode Cloud exécute le travail dans un environnement distant configuré.

Les modes Local et Worktree s'exécutent sur l'ordinateur de l'utilisateur. Le mode Worktree fournit une isolation Git entre le travail courant et les changements produits par Codex.

3. Modèles, raisonnement et contexte

3.1 Modèles

Codex utilisé des modèles OpenAI adaptés aux tâches de développement. La documentation officielle indiqué gpt-5.5 comme modèle recommandé pour la plupart des tâches Codex, avec un accent sur le codage complexe, l'utilisation d'outils, la planification et les travaux multi-étapes.

La disponibilité des modèles dépend du plan, de la surface et des droits du compte.

3.2 Raisonnement

Codex peut décomposer une demande, établir un plan, lire les fichiers pertinents, effectuer des modifications, lancer des vérifications et ajuster son travail selon les résultats obtenus.

Le raisonnement est contraint par le contexte accessible, les instructions système et utilisateur, les règles du projet, les outils disponibles, les permissions, le bac à sable et la politique d'approbation.

3.3 Contexte de conversation

Un fil Codex conserve les messages, decisions, plans et résultats pertinents pour le travail en cours. Les sessions peuvent être reprises lorsque la surface le permet.

Le contexte peut comprendre le prompt courant, les messages précédents, le contenu de fichiers lus, les sorties de commandes, les images ou documents fournis, les consignes de projet et les données d'outils connectes.

3.4 Mémoires

Lorsque la fonction est disponible, les mémoires transportent des informations stables entre fils, telles que préférences, conventions de projet ou habitudes récurrentes. Elles complètent les consignes de projet mais ne remplacent pas les instructions explicites du fil.

4. Outils et opérations sur le code

4.1 Lecture et modification de fichiers

Codex peut lire les fichiers accessibles, analyser leur structure et produire des changements. Les modifications sont limitées par l'espace de travail, le bac à sable et les permissions accordées.

4.2 Exécution de commandes

Codex peut exécuter des commandes via la surface active, notamment pour inspecter un projet, lancer des tests, démarrer un serveur local, formater du code ou vérifier un résultat. L'exécution est soumise aux approbations et aux restrictions du bac à sable.

4.3 Revue de code

Codex fournit une fonction de revue qui recherche anomaliés, regressions, risques de comportement et lacunes de test. Dans la Codex App, le panneau de diff permet d'examiner les changements, de commenter des lignes et de demander des corrections.

4.4 Git

Codex intégré ou utilisé des fonctions Git selon la surface : affichage de diff, staging, commit, push, création de pull request, travail en worktree et inspection de l'état du dépôt.

4.5 Terminal intégré

Dans la Codex App, chaque fil dispose d'un terminal associé au projet ou au worktree. Ce terminal fournit un canal local pour les commandes de vérification et d'exploitation du projet.

4.6 Navigateur intégré et Browser Use

La Codex App fournit un navigateur intégré pour prévisualiser, commenter et vérifier des pages locales, des previews de fichiers ou des pages publiques ne nécessitant pas de connexion.

Browser Use permet à Codex d'opérer certaines pages, notamment des serveurs de développement locaux et des previews de fichiers, lorsque les permissions et plugins requis sont présents.

4.7 Computer Use

Computer Use permet à Codex d'interagir avec des applications macOS ou Windows en voyant, cliquant et tapant. Cette fonction vise les tests d'applications de bureau, la reproduction d'anomaliés GUI et les flux non accessibles par API ou plugin.

4.8 Entrées image et génération d'images

Codex accepte des images comme contexte. Il peut aussi générer ou modifier des images dans un fil lorsque la fonction est disponible, notamment pour des assets d'interface, illustrations, placeholders ou sprites.

4.9 Artefacts non-code

Codex peut produire et vérifier des artefacts tels que PDF, documents, feuilles de calcul et présentations. La Codex App peut affichér ces artefacts dans la barre laterale de tâche.

5. Personnalisation

5.1 AGENTS.md

AGENTS.md fournit des consignes durables pour un dépôt ou une partie de dépôt. Les fichiers les plus proches du répertoire de travail ont priorité sur les fichiers plus généraux.

AGENTS.md sert à décrire les commandes de build et de test, conventions de code, attentes de revue, chemins ou fichiers à privilégier et instructions propres à une arborescence.

5.2 Configuration

Codex peut être configuré par fichiers de configuration, variables d'environnement et réglages de surface. La configuration couvre notamment modèles, approbations, bac à sable, outils, MCP, hooks et préférences d'exécution.

5.3 Skills

Les skills sont des capacités réutilisables, définies par un dossier contenant un fichier SKILL.md et, éventuellement, des scripts, références et assets.

Elles permettent d'encapsuler des workflows répétables, une expertise de domaine, des modèles et ressources, des procédures avec validations et des intégrations indirectes via MCP.

5.4 Plugins

Les plugins sont des unités distribuables qui peuvent inclure skills, outils, serveurs MCP, hooks, assets ou applications. Ils servent à empaqueter des capacités partageables.

5.5 MCP

Le Model Context Protocol relié Codex à des outils et systèmes externes. Les configurations MCP sont partagees entre la Codex App, le CLI et l'extension IDE lorsque la surface les prend en charge.

5.6 Hooks

Les hooks permettent d'associér des traitements à des événements du cyclé de vie Codex. Ils servent à appliquer ou vérifier des politiques techniques autour des opérations de l'agent.

5.7 Subagents

Les subagents représentent des agents spécialisés auxquels Codex peut déléguer des parties de travail. Ils sont destines aux tâches qui beneficient d'une expertise ou d'un contexte spécialisé.

6. Sécurité, bac à sable et approbations

6.1 Bac à sable

Le bac à sable définit les ressources accessibles à Codex : fichiers lisibles, emplacements modifiables, accès réseau et autrès capacités d'exécution. Par défaut, Codex limite son travail au projet actif lorsque la surface le permet.

6.2 Approbations

Les approbations déterminent les opérations qui exigent une confirmation de l'utilisateur ou une revue automatique. Elles peuvent être accordees pour une action unique ou pour une portee plus large selon la surface et la politique.

6.3 Permissions

Les permissions contrôlent l'accès aux fichiers, commandes, réseaux, applications locales et outils externes. Les permissions peuvent provenir de la configuration locale, de la politique admin ou d'une approbation utilisateur.

6.4 Réseau

L'accès réseau peut être désactivé, limité, mis en caché ou accorde selon le mode. La recherche web de premiere partie est disponible dans certaines surfaces et peut fonctionner via caché ou résultats en direct selon la configuration.

7. Automatisation et intégrations

7.1 Automations

Les automations exécutent des tâches récurrentes ou planifiees. Elles peuvent produire rapports, vérifier erreurs, surveiller changements ou continuer un suivi.

Les automations de fil conservent le contexte d'une conversation particuliére. Les automations de projet ou autonomes lancent des travaux récurrents avec un contexte de départ défini.

7.2 Mode non interactif

Le mode non interactif permet d'exécuter Codex dans des scripts ou chaînes d'intégration. Il convient aux tâches définies par un prompt et une configuration.

7.3 Codex SDK

Le SDK Codex fournit une interface de programmation pour intégrer Codex dans des workflows applicatifs ou outils internes.

7.4 App Server et mode distant

L'app server permet à un cliént Codex de se connecter à un serveur distant par WebSocket ou socket local, avec mécanismes d'authentification appropriés.

7.5 Intégrations GitHub, Slack et Linear

Les intégrations reliént Codex à des systèmes de collaboration et de suivi. Leur disponibilité dépend de l'installation, des autorisations de compte et de la configuration d'organisation.

8. Objets de référence

8.1 Projet

Un projet associé à Codex à un répertoire ou dépôt. Il définit le périmètre de travail principal.

8.2 Fil

Un fil est une conversation opérationnelle avec Codex. Il contient les demandes, réponses, plans, sorties et modifications associées à une tâche ou serie de tâches.

8.3 Worktree

Un worktree est une copie de travail Git séparée, utilisée pour isoler les modifications d'un fil par rapport au répertoire principal.

8.4 Chat

Un chat est un fil qui ne dépend pas nécessairement d'un dossier projet ou dépôt. Il convient à la recherche, la planification et les workflows d'outils connectes.

8.5 Skill

Une skill est un module d'instructions et de ressources réutilisables sélectionné par Codex lorsqu'une tâche correspond à sà description.

8.6 Plugin

Un plugin est un paquet installable qui regroupe des capacités Codex.

8.7 MCP server

Un serveur MCP expose des outils, ressources ou actions externes à Codex.

8.8 Hook

Un hook est un traitement associé à un événement Codex.

8.9 Automation

Une automation est une exécution planifiée ou récurrente d'une instruction Codex.

8.10 Bac à sable

Le bac à sable est l'ensemble des limites d'accès appliquées aux actions de Codex.

9. Entrées de commande et de configuration

Ce chapitre décrit les principales entrées fonctionnelles. Il ne constitue pas une procédure d'utilisation.

9.1 Commande codex

La commande codex représente le point d'entrée principal du cliént terminal. Elle ouvre une session interactive lorsque lancee sans sous-commande et accepte un prompt initial lorsque du texte lui est fourni.

Fonctions associées :

  • initialisation d'une conversation dans le répertoire courant ;
  • sélection d'un modèle ou d'options de session ;
  • interaction avec le bac à sable et les approbations ;
  • affichage de plans, diffs et sorties ;
  • reprise ou délégation à des sous-commandes.

9.2 Sous-commande résumé

résumé représente la fonction de reprise de session. Elle rattache une nouvelle exécution à un transcript existant et conserve les éléments de contexte pertinents de la conversation précédente.

Les variantes de reprise distinguent la dernière session, une session choisie dans une liste, une session spécifique par identifiant et, selon l'option, les sessions hors du répertoire courant.

9.3 Sous-commande exec

exec représente l'exécution non interactive. Elle traite une instruction fournie au lancement et produit une sortie destinée aux scripts, chaînes d'intégration ou tâches reproductibles.

Cette fonction conserve les contraintes de sécurité de Codex : bac à sable, approbations, configuration, instructions de projet et limites d'outils.

9.4 Sous-commandes serveur

Les fonctions serveur, notamment l'app server et les modes distants, exposent une session Codex à un cliént se connectant par canal local ou WebSocket. Elles fournissent une séparation entre le processus de service et le cliént d'interaction.

Les fonctions serveur comprennent :

  • écoute sur une adresse locale ou distante ;
  • authentification par jeton ou mécanisme configure ;
  • connexion de cliénts distants ;
  • contrôle d'une session depuis une autre surface.

9.5 Commandes slash

Les commandes slash représentent des actions de session disponibles dans les interfaces interactives. Elles exposent des fonctions telles que l'affichage d'état, le changement de modèle, la gestion de contexte, la sortie de session ou l'accès à l'aide.

La disponibilité exacte varie selon CLI, IDE et App.

9.6 Fichier AGENTS.md

AGENTS.md est un fichier d'instructions durables. Il est lu comme guidance de projet et peut exister à plusieurs niveaux : global utilisateur, racine du dépôt et sous-répertoires. Les fichiers plus proches du répertoire de travail définissent les consignes les plus spécifiques.

Attributs fonctionnels :

  • portee durable ;
  • versionnable avec le dépôt lorsqu'il est place dans le projet ;
  • priorité par proximité ;
  • adaptation du comportement de Codex avant le travail ;
  • support des conventions, commandes et attentes de revue.

9.7 Fichier de configuration

Le fichier de configuration Codex représente les réglages persistants du cliént et de l'agent. Il peut definir les valeurs par défaut relatives au modèle, aux permissions, au bac à sable, aux serveurs MCP, aux hooks et aux préférences d'exécution.

Les réglages effectifs résultent de la combinaison entre configuration globale, configuration projet, options de lancement, politiques gérées et choix d'interface.

9.8 Variables d'environnement

Les variables d'environnement fournissent un canal de configuration externe au fichier de configuration. Elles conviennent aux secrets, aux chemins propres à l'environnement et aux réglages injectés par une chaîne d'exécution.

Une variable d'environnement peut être ignorée ou surclassée si la surface, la politique d'organisation ou la configuration active impose une autre valeur.

9.9 Règles et hooks

Les règles définissent des politiques d'autorisation ou de refus autour des commandes et actions. Les hooks fournissent des points d'exécution autour d'événements Codex.

Les règles sont déclaratives. Les hooks sont proceduraux. Les deux mécanismes complètent les approbations utilisateur et les limites du bac à sable.

9.10 Profils de permissions

Les profils de permissions représentent des ensembles de capacités accordées ou refusées : lecture, écriture, exécution, accès réseau, accès à des répertoires supplémentaires ou utilisation d'outils externes.

Les profils peuvent être définis localement, gérés par une organisation ou demandés pendant une session.

10. Matrice fonctionnelle

10.1 Fonctions par surface

La Codex App couvre les fils paralleles, worktrees, terminal intégré, diffs Git, artefacts, navigateur intégré, automations, notifications et synchronisation IDE.

Le CLI couvre l'interaction terminale, les sessions scriptables, la reprise de conversation, les modes non interactifs, les options de lancement et les modes serveur ou distant.

L'extension IDE couvre le contexte de l'éditeur, les fichiers ouverts, les sélections, les commandes IDE et l'iteration directement dans l'environnement de développement.

Codex Web et Cloud couvrent l'exécution distante, les environnements configures, les tâches déléguées et les intégrations cloud.

10.2 Fonctions par type de tâche

Pour compréhension de code, Codex utilisé lecture de fichiers, analyse de structure, synthèse et explication.

Pour modification de code, Codex utilisé planification, édition, exécution de commandes, vérification et résumé.

Pour revue, Codex utilisé inspection de diff, recherche de bugs, risques, regressions et tests manquants.

Pour automatisation, Codex utilisé prompts récurrents, contexte de projet, worktrees ou environnement local, skills et notifications.

Pour intégration externe, Codex utilisé MCP, plugins, connecteurs, API ou surfaces cloud selon la configuration.

10.3 Relations entre mécanismes

AGENTS.md décrit les règles durables du dépôt. Les skills décrivent des workflows réutilisables. Les plugins empaquetent des capacités. MCP connecte Codex à des systèmes externes. Les hooks imposent des traitements autour d'événements. Les automations planifient l'exécution.

Ces mécanismes sont complementaires : AGENTS.md oriente le comportement, les skills decrivent comment realiser des workflows, MCP donne accès à des données ou actions, les hooks encadrent l'exécution et les automations declénchent le travail dans le temps.
